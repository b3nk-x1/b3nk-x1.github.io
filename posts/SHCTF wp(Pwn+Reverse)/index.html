[阶段1] Linklist

看一下附件，题目实现了一个简单的单向链表，支持：

create(size, content)：创建节点（节点结构体 + 内容缓冲区）

delete()：删除头结点

show()：打印头结点内容（printf(“%s”)）

edit()：编辑头结点内容（固定读入 0x20 字节）

核心漏洞在 edit()：无视 create 时申请的 size，始终向 content写 0x20 字节，导致堆溢出

每个节点大致为：

malloc(0x10) 申请 node 结构体

malloc(size)`申请 content

read(0, content, size)写入用户数据

关键是：

无论 content 实际分配多大，都固定写 0x20。

在 glibc 堆中，malloc(0x18) 对应 chunk size 为 0x20，而可写区域通常为 0x18。因此：

create 时写 0x18不溢出

edit 时写 0x20溢出 8 字节

这 8 字节恰好可以覆盖后一个 chunk 的 size 字段

我们利用这 8 字节做 chunk size overwrite，进而制造 chunk overlap

思路：tcache + size overwrite → 伪造 0x40 chunk → overlap → 篡改节点指针

利用 tcache 的特性：

0x20 size 的 chunk 会进 tcache(0x20)

如果能把某块 0x20 chunk 的 size 改成 0x41（0x40 | PREV_INUSE），它在 free 时会进入 tcache(0x40)

再 malloc(0x30) 时会取出该“伪造的 0x40 chunk”作为 content，从而和其它对象发生重叠

通过重叠，我们能在写 content 的同时覆盖 node 结构体字段（特别是 content 指针），把它改成任意地址。

随后：

show() 变成“从任意地址读（打印）

edit() 变成“向任意地址写（固定最多 0x20）

我们每一轮都构造一个“受控头节点”，让它的 head->content 指向任意地址。

1.create(0x18) 得到节点 A

2.create(0x18) 得到节点 B（B 在链表头）

此时堆上每个节点会分配：

B.struct (0x20 chunk)

B.content (0x20 chunk)

A.struct (0x20 chunk)

A.content (0x20 chunk)
（顺序视实现，但关键是它们大概率紧邻）

  1. delete() 删除 B
    B 的 struct 与 content 两个 0x20 chunk 都进入 tcache(0x20)
  2. edit(A) 触发溢出
    对 A.content 写 0x20 字节，溢出 8 字节到下一个 chunk 的元数据，把 B.struct chunk 的 size 改为 0x41（伪装成 0x40 chunk）
  3. create(0x18) 申请一个节点 C
    会从 tcache(0x20) 取回被“改过 size 的那块”作为某个分配（通常能复用到那块 B.struct chunk）
  4. delete() 删除 C
    free(C.struct) 时，glibc 看到 chunk size=0x40，于是它进了 tcache(0x40)
  5. create(0x30) 申请节点 D
  • D.content 需要 0x40 chunk（0x30 对应 chunk size 0x40），会从 tcache(0x40) 取出那块“伪造的 0x40 chunk”
  • 由于这块 chunk 实际上覆盖了原先相邻的两块 0x20 区域，会造成 D.content 与某个 node.struct 重叠

在 create(0x30) 输入 D.content 时，我们就可以覆盖到 D.struct 中的字段，把 D.content 指针改成我们指定的地址 target

至此得到“受控头节点”：head->content = target

[阶段1] execve?orw?

题目名看起来像 “execve / orw”，但实际上程序在执行我们输入的 shellcode 前开启了 seccomp 沙箱，把可用 syscall 限死到几乎只剩 exit，导致：

不能 open/read/write

不能 execve

无法直接输出 flag

因此需要换思路：利用“程序行为差异”进行侧信道泄露

对 main 反汇编可得到程序流程：

  1. 打开 flag 文件

  2. mmap(0x11451000, 0x1000, PROT_READ|PROT_WRITE, MAP_PRIVATE, fd, 0)

    flag 内容被映射到固定内存地址 0x11451000 开始

  3. read(0, 0x11451500, 0x500)

    我们发送的 payload（shellcode）会被读到 0x11451500

  4. sandbox() 开启 seccomp

  5. call *(0x11451500) 直接跳转执行 shellcode

也就是说：flag 在内存里、shellcode 也在内存里，但 syscall 被锁死。

sandbox() 逻辑等价于：

seccomp_init(SCMP_ACT_KILL)：默认行为 kill

seccomp_rule_add(ctx, SCMP_ACT_ALLOW, 60, 0)：只允许 syscall 60（exit）

seccomp_load(ctx)

结论：shellcode 只能做纯用户态计算，最终最多调用 exit() 结束。

虽然不能 write 输出 flag，但我们可以让程序在两种分支上表现出明显时间差：

若猜测的字节 正确：立刻 exit(0) → 连接很快断开

若猜测的字节 错误：先执行一段忙等循环（纯计算，无 syscall）再 exit(1) 连接明显更晚断开

客户端用“连接关闭耗时”判断这一位是否猜对，从而逐字节爆破整段 flag

[阶段2] 整数面

在 .text 里能看到典型流程：

  1. fgets(buf, 0x9e, stdin)
  2. 调 advapi32!SystemFunction033（RC4 类）对前 0x96=150 字节处理
  3. 每字节做一次“翻倍 + 条件 XOR”
  4. 做一个变种 Base64（带 rolling acc）编码成 200 字节
  5. 与 .rdata 中 200 字节常量比较（比较前对常量逐字节取 NOT）

关键：第 4 步输出看起来“像 base64”，但不是标准 base64，不能直接 base64.b64decode()

编码逻辑等价于：

先按标准 base64 的 bit-split 得到 idx（0..63）

但输出前做：acc = (acc + idx) % 64，输出 alphabet[acc]

逆过来就是：

取输出字符 c 的索引 v

idx = (v - acc_prev) mod 64

更新 acc_prev = v

这样可恢复标准 base64 的 idx 流，再 4->3 还原 150 字节数据。

IDA找函数 0x14000161f，它做了三件事：

先填 S[i]=i，然后做固定三轮 swap（用 0x140003470 交换两个字节）。

用 S-box 替换 .data:0x140004120 的 0x34 字节即：buf[i] = S[buf[i]]

替换完成后：

前 0x28(40) 字节：是一段可读字符串（这里就是 flag 后半段，且包含 }）

然后它立刻 memset(buf,0,0x28) 把这 40 字节清掉（所以运行结束你再看内存就没了）

最后 12 字节：被当作索引映射到 base64 alphabet，得到 BV1GJ411x7h7，用来覆盖 secret key 前 12 字节**

[阶段3] strange_chain

IDA检查一下逻辑：

  1. 读取一行输入（去掉 \r\n）
  2. 对输入做 PKCS#7 padding 到 8 字节对齐
  3. 要求 padding 后长度必须为 0x28（40 字节）
  4. 将 40 字节分为 5 个 8-byte block 做一套“奇怪的函数链变换”
  5. 结果与 .rdata 中固定 40 字节常量做 memcmp
  6. 相等即通过

因此 flag 的明文长度必须在 32~39 字节之间（因为 pad 后固定 40）
根据题目提示这题不是传统循环，而是构造一段类似 trampoline 的代码链：

mov rax, <func_ptr>

call rax

mov rax, <next_stub>

jmp rax

这样把一堆 tiny gadget 串成“链”，静态看起来像无意义跳转，但语义等价于一个固定的轮函数。

分组方式：

40 字节 → 5 组，每组 8 字节

每组拆成两个 uint32：v0, v1（little-endian）

每组进行 12 轮更新（轮计数 ctr = 1..12）

轮函数还原：12 轮 + swap 条件

每轮会用到两张表：

key[0..25]：26 个 32-bit key word（运行时初始化生成）

ba0[0..13]：14 个 32-bit 轮常量（由 .rdata 的 16 字节 seed 派生）

轮函数核心是一个 “g(x)”：

g(x) = (ROL(x,13) ^ ROL(x,5)) + x （所有运算按 uint32 溢出截断）

然后每轮（ctr = 1..12）：

  1. v0 ^= g(v1)
  2. v0 = ROL(v0, (v1 ^ ba0[ctr]) & 0x1f) 或（另一变体：先 add 后 rol）
  3. v0 += key[2*ctr]
  4. v1 ^= g(v0)
  5. v1 = ROL(v1, (v0 ^ ba0[ctr]) & 0x1f) 或（另一变体：先 add 后 rol）
  6. v1 += key[2*ctr+1]
  7. 若 ctr % 3 == 0，则 swap(v0, v1)

此外还有开头的两步 whitening：

v0 += (key[0] ^ ba0[0])

v1 += (key[1] ^ ba0[0])

因为只用了 XOR / ADD / ROL / swap（都可逆），所以该分组算法整体可逆，直接写 decrypt_block() 就能把 memcmp 常量反推回明文

程序 .rdata 里有 16 字节 seed，初始化时通过一个混合函数生成 ba0[0..13]。

更阴间的地方是：key[0..25] 不是写死的，而是通过另一条 gadget chain 初始化：

先设定 key[0] = 0x740EB8B8

再用递推生成一段 key（key[i] = key[i-1] - 0x4432330F）

然后循环 78 次执行若干 gadget（更新 key[26..] / 更新状态寄存器）

最终得到 key[0..25] 供加密轮函数使用

实际样本中存在两条链的排列变体（init_variant / cipher_variant），但可以直接在脚本里全枚举并用 PKCS#7 过滤正确解。

这也是我最初遇到的坑：如果 PE 的内存映射不正确（把 .bss 当成文件内容去读），key 初值会错，导致解密结果不满足 padding。正确做法是构造虚拟内存镜像：只拷贝各节 RawSize，其余默认 0

从 trace 中能直接观察到每个操作的输出规律（尤其是对一些简单输入如 0/1/2 的表现），可以确定：

op = 0：32-bit 加法（uint32 溢出截断）

trace 里能看到它用位运算实现加法（循环进位），但语义就是 (a + b) mod 2^32

op = 1：XOR

实现形如 (a & ~b) | (~a & b)，等价 a ^ b

op = 2：左移 b 位

循环调用“左移 1”的小函数 b 次

op = 3：右移 b 位

同理循环右移 1

op = 4：乘法

典型移位加法实现，等价 (a * b) mod 2^32

因此后续直接用标准算术并在每步 & 0xffffffff 即可复现

trace 中给了两枚种子：

seed1 = 0x12345678

seed2 = 0xDEADBEEF

并给出 key 计算过程（全部按 uint32 截断）：

k0 = (seed1 & 0xFFFF) * 0x1337

k1 = seed2 + 0xAAAA

k2 = k0 ^ k1

k3 = k2 * 2 + 1

从 trace 中每轮重复出现的模式可识别为 TEA 的 Feistel 结构（32 轮，sum += delta），但移位常量被改了：

delta = 0x9E3779B9

标准 TEA 的 (v<<4) / (v>>5) 被改为：
(v<<2) 和 (v>>4)

每轮更新：

  1. sum += delta
  2. v0 += ((v1<<2)+k3) ^ (v1+sum) ^ ((v1>>4)+k1)
  3. v1 += ((v0<<2)+k2) ^ (v0+sum) ^ ((v0>>4)+k0)

其中 v0, v1, sum 全部是 uint32。

trace 中给了一个 40 字节 target。程序按 little-endian 拆成 10 个 uint32：

共 5 个 64-bit block

每块是 (v0, v1) 两个 32-bit

然后对每块执行上述 TEA 变体加密/解密。对每块做 32 轮逆运算，拼回 40 字节，得到原始输入字符串。

TEA 类结构解密写法固定：

初始 sum = delta * 32 (mod 2^32)

每轮：

  1. v1 -= F(v0, sum)
  2. v0 -= F(v1, sum)
  3. sum -= delta

对应本题的 F：

F_for_v0(v1, sum) = ((v1<<2)+k3) ^ (v1+sum) ^ ((v1>>4)+k1)

F_for_v1(v0, sum) = ((v0<<2)+k2) ^ (v0+sum) ^ ((v0>>4)+k0)
