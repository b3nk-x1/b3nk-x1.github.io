题目给出了 n、e、密文 c,以及 p 高 256 位加密后的 c_hint。服务还提供了 256 次最低位解密 oracle,每次输入一个密文,返回对应明文的最低位。核心思路是先利用 RSA 的乘法同态,把 c_hint 乘上 2^(e*k),这样解密后就等价于观察 p_high * 2^k mod n 的最低位。由于 p_high < 2^256,而 n 是 1024 位,前面大量二进制位天然为 0,所以 256 次查询刚好足够把 p_high 唯一恢复出来,并验证满足 pow(p_high, e, n) == c_hint。接着设 p = (p_high << 256) + (b << 248) + x,其中中间的 8 位 b 暴力枚举,低 248 位 x 视为小根。对每个 b 构造关于 x 的一元多项式格,用 fplll 做 LLL 规约,再从规约后的短向量多项式里找整数根;一旦找到满足 n % p == 0 的候选,就成功分解出 n = p * q。最后计算 phi = (p-1)(q-1),求出私钥 d = e^{-1} mod phi,解密 c 得到最终 flag:H&NCTF{rsa_pro_8ab138d3c13c_d32a149e-11ce-4a98-aa86-66ffd0521fa8}。

EZVM

先给出最终 flag:H&NCTF{ez_wbaes_vm_2026}。

这题是一个 APK 逆向题。先用 JADX-GUI 查看 Java 层逻辑,再用 IDA 分析 libezvm.so。Java 层没有真正的校验逻辑,只是在按钮点击后读取输入,然后调用 native 方法 checkFlag(String),所以关键点在 so 里。

先看 Java 层,在 MainActivity 中能定位到程序最终调用的是 native 的 checkFlag。结合 System.loadLibrary 可以确定对应的 so 是 libezvm.so。这一层只能确定入口,真正的 flag 校验并不在 Java 里。

把 libezvm.so 丢进 IDA 后继续跟进。可以发现 checkFlag 不是普通导出函数,而是通过 RegisterNatives 动态注册的。沿着注册表继续看,可以确认 checkFlag(Ljava/lang/String;)Z 对应到 native 里的实际校验函数。

继续分析这个函数,能看到它先取输入字符串和长度,然后把数据送进一个内部解释器执行。也就是说这题不是普通 JNI 字符串比较,而是把核心算法塞进了一个自定义 VM 里。VM 最后返回 1 表示校验通过,返回 0 表示失败。

继续还原 VM 逻辑后,可以确定整体流程是把输入做一轮 32 字节变换,再和程序内置结果比较。从 so 中可以提取出目标密文:

继续跟内部子 VM,可以看出这一层结构和 AES 很像,但不是标准 AES,更像做了白盒处理后的分组变换。里面有查表、轮变换、字节置换,输入会分成两个 16 字节块处理,不足时还会补位。所以这里最稳的做法不是纯手推,而是把 VM 的指令语义补出来,直接写脚本模拟真实执行流程。

  5. 去掉补位后得到原始 flag;

Pwn

notezpz2

这题的核心漏洞是 delete_question() 在执行 free(q->text); free(q); 之后,没有把全局数组 questions[id] 置空,导致程序保留了一个悬空指针,形成 Use-After-Free。后续如果继续对这个已经删除的题目执行 ask 或 set,程序仍然会访问已经释放的堆块。

首先通过 C -> C -> D(0) -> C -> A(0) 这组操作制造重叠关系,让旧的题目指针指向新分配的对象,并从已经进入 unsorted bin 的堆块元数据中泄漏 libc 地址。根据泄漏值可以计算出远端 libc 基址。本题远端环境实际对应的是 glibc 2.39,因此利用时需要使用这一版 libc 的偏移。

接着继续大量申请和释放不同大小的 question 结构体块与 text 块。由于结构体块大小约为 0x530,文本块大小约为 0x420,这些块在释放后会进入 unsorted bin / largebin。通过精心布置多个 largebin chunk,可以让 stale question->text 不再指向原来的文本区,而是被 largebin 元数据覆盖成一个可控的 chunk header。这样在后续调用 set_question() 时,fgets(q->text, ...) 的目标地址就可以被伪造成:

从而把一次普通写入转化为任意地址写。

由于题目环境开启了 SHSTK / CET,直接覆盖返回地址做传统 ROP 会触发保护并崩溃,所以最终采用 FSOP。具体做法是在 libc 的 .bss 区域伪造一个 _IO_FILE 结构,再伪造对应的 _wide_data 和 wide vtable,把 fake_vtable->__doallocate 指向 system,然后把 _IO_list_all 改写为 fake FILE 的地址。最后向菜单发送 Q,程序正常退出时会触发 _IO_flush_all(),从而走到伪造的 wide vtable,最终执行 system("exec sh")。
