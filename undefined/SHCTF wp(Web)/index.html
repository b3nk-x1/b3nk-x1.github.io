访问靶机后，highlight_file(FILE)输出源码，核心是一个 PHP 反序列化漏洞，需要构造 POP 链读取 /flag。

源码关键部分

触发链：

Solar::__call() 中可以实例化任意类并调用任意方法。利用 PHP 原生类 SplFileObject 打开flag 文件，再用 fpassthru()`方法将文件内容直接输出。

$func = “SplFileObject”，$args[0] = “/flag”→new SplFileObject(“/flag”)

$road = “fpassthru”→$exploar->fpassthru()输出文件内容

/flag 不匹配正则黑名单；fpassthru虽然包含 passthru 但黑名单只检查 $args[0]，不检查方法名

反序列化后紧跟 throw new Exception(“How to Travel?”)，正常情况下异常抛出会阻止 __destruct在脚本结束时执行

通过数组重复键触发 GC 提前回收

将 Sun 对象放入数组，然后修改序列化字符串使数组中出现重复的键 i:0：

第二个 i:0会覆盖第一个，Sun 对象的引用计数降为 0，PHP 的垃圾回收器会在反序列化期间立马销毁该对象，从而触发 __destruct——这发生在 throw 之前。

Solar 类有一个 private $Sun属性，PHP 序列化 private 属性时键名格式为 \x00ClassName\x00PropertyName，即 \x00Solar\x00Sun

直接在 HTTP 请求中传递 \x00可能被截断。使用 PHP 序列化的 S:格式代替 s 来正确表示空字节：

Solar 类共有 9 个属性1 个 private + 8 个 public），序列化头必须写 O:5:”Solar”:9:{…}，而不是 8。属性计数错误会导致 unserialize() 直接返回 false。

由于余额变为负数，满足了某种异常条件，即可在 /flag 获取 flag。
